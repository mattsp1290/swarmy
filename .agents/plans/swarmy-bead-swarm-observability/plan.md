- Description: Swarmy is a Nim/Svelte monorepo for observing and coordinating concurrent `/goal /bead-swarm` runs. A Nim binary, shaped similarly to `~/git/advisor`, provides a Jazzy-backed HTTP API, an MCP stdio surface, and CLI commands. Agents run `swarmy init` to create a persistent `./.swarmy/` identity, then record swarm events through MCP tools or equivalent CLI commands. The Svelte app shows multiple simultaneous swarms, active beads, assigned agents, and whether each bead is in coding, validation, review, merge, blocked, or complete stages.
- Technical Stack: Nim 2.2+, Nimble, Jazzy Framework backend, SQLite persistence via Jazzy DB where practical, NimCP or a thin JSON-RPC MCP implementation for MCP tools/resources/prompts, Svelte/SvelteKit frontend, Node/npm tooling, Beads `bd` CLI integration for read-side reconstruction, monorepo layout.
- Specific Requirements: No user-specified auth/perf/accessibility integrations beyond production maturity. Product requirements: mirror/reconstruct from `bd` where reliable; track additional swarm state where Beads is insufficient; support multiple concurrent swarm runs; persist across process restarts; expose MCP tools and CLI equivalents; serve `/bead-swarm` as an MCP resource or prompt; record ecosystem gaps as upstream-style requests in the owning project request folder. Initial research notes and external request files have already been written for Jazzy realtime support and NimCP MCP-version clarity.
- Out of Scope: No explicit user non-goals. For the first version, do not block on first-class Jazzy realtime support or full current-spec NimCP upstream fixes; file requests and ship with REST polling plus clear seams. Do not implement cloud multi-user hosting, distributed multi-node fanout, billing, or non-Beads issue trackers unless a later requirement adds them.
- Maturity Target: production. Include focused CI/build/test gates, persistence/restart coverage, validation, local auth/token defaults, structured logs, runbook documentation, and packaging. Observability is core product behavior, not optional polish.
- Definition of Done: A user can run `/goal /bead-swarm`, the agent is instructed to run `swarmy init`, agent activity is recorded through Swarmy, and opening the Svelte app shows active runs, active beads, assigned agents, and each bead's current stage, including coding and review. No hard deadline given.
- Edge cases surfaced: Swarmy is a buddy for coding agents, not the source of truth for Beads. Use `bd` snapshots/read-only reconstruction for canonical bead status, and append Swarmy events for agent/stage state. A swarm run is identified by `./.swarmy/` metadata generated by `swarmy init`; include canonical repo path, run ID, timestamps, and config. State must persist across restarts. Agent write surfaces must exist as both MCP tools and CLI equivalents, including a CLI equivalent for the `/bead-swarm` instructions. Multiple swarms may run simultaneously across repos or sessions, so APIs and UI must filter by run ID, use run-scoped cursors, and tolerate concurrent writers. Defaults: SQLite store; integer surrogate row IDs; generated run UUID/string IDs; RFC3339 text timestamps; append-only event log; full-replace run metadata update; `--db` / `SWARMY_DB` config fallback; REST polling before realtime; safe default bind address; explicit stage enum for `coding`, `validation`, `review`, `merge`, `blocked`, `complete`, and `unknown`.
- Chosen labels: `setup`, `core`, `data`, `agent`, `api`, `ui`, `verify`, `docs`, `ops`.
