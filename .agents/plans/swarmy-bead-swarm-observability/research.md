## User-Confirmed Product Shape
- Swarmy is a Nim/Svelte monorepo for making concurrent `/goal /bead-swarm` runs visible.
- The backend should be Nim using Jazzy Framework.
- The frontend should be Svelte.
- `swarmy init` should write a unique run ID and metadata into `./.swarmy/`.
- Swarmy should mirror/reconstruct from `bd` where that is reliable, and store extra agent progress where Beads does not have enough detail.
- Agents should be able to write through MCP tools and equivalent CLI commands.
- The `swarmy` binary should serve a `/bead-swarm` MCP resource or prompt so agents can discover the workflow.
- First milestone: run `/goal /bead-swarm`, have agents record state, and view active beads plus coding/review stage in the Svelte UI.

## Web Research
- Jazzy Framework is a Nim REST framework with routing, middleware, validation, SQLite, auth helpers, config loading, and a dev UI. The public site describes built-in JWT auth, validation, SQLite support, scaffolding CLI, async middleware, and JSON error responses.
- Jazzy release `v0.4.0` was published on 2026-06-21. It added Melody template features, form parsing, database pagination, automatic timestamps, soft deletes, and route grouping fixes.
- Jazzy's current repo has no obvious first-class WebSocket/SSE route helper in the public routing API. Its own `SUGGESTED_ISSUES.md` already lists database transactions, migration tooling, and real-time log viewer as future work.
- Nimble can manage per-project Nim versions from `.nimble` requirements and pin exact versions in `nimble.lock`; use this for reproducible Swarmy builds.
- SvelteKit is the practical frontend shell if the app needs routing, build optimization, SSR/static choices, and modern app conventions. Plain Svelte remains the component model.
- MCP 2025-06-18 defines servers as exposing resources, prompts, and tools. Resources share data/context; prompts expose user-invoked workflow templates; tools are model-controlled functions.
- MCP resources support templates and subscriptions; tools require a declared `tools` capability and should be surfaced with clear user trust/consent boundaries.
- NimCP exists and supports tools, resources, prompts, stdio, HTTP, WebSocket, and streamable HTTP transport. Its README still advertises full MCP `2024-11-05` support and its code sets `MCP_PROTOCOL_VERSION = "2024-11-05"`, while `STREAMABLE.md` claims 2025-06-18 streamable HTTP compliance. That mismatch should be upstreamed before Swarmy depends on current-spec behavior too deeply.
- `treeform/ws` is an available Nim WebSocket library, but for Swarmy v1 the plan should not block on it; REST polling is sufficient for the first UI, with a clean service seam for future push updates.

## Planning Defaults
- Store: SQLite via Jazzy's database layer unless a specific blocker appears.
- IDs: Swarmy run IDs are generated by `swarmy init`; relational rows may use integer surrogate IDs internally.
- Timestamps: store `created_at` and event times as RFC3339 text.
- Updates: agent progress is append-only events where possible; run metadata can use full-replace update semantics.
- Realtime: start with REST polling and event sequence cursors; keep an interface seam for SSE/WebSocket.
- Config: single CLI flag with env fallback, e.g. `--db` / `SWARMY_DB`, plus `--repo` when operating outside cwd.
- Security: production target means local auth/token controls, payload validation, and a safe default bind address.
